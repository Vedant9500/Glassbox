# Curve Classifier End-to-End Algorithm

This document describes the full pipeline from synthetic data generation to final operator prediction and ONN warm-start / fast-path regression.

---

## 1) Adversarial Data Generation & Training Stage

Primary script: `scripts/train_adversarial_classifier.py`
Secondary script: `scripts/teacher_generator.py`

Instead of generating data from static templates, the system now uses **Adversarial Self-Play** to dynamic train the classifier.

### 1.1 The Two Agents

1. **The Teacher (TeacherRNN):**
   - An autoregressive RNN trained via REINFORCE (Policy Gradient).
   - Generates a sequence of tokens (`+`, `*`, `sin`, `x`, `1.0`, `pow`, etc.) in prefix notation.
   - Converts the prefix tokens into an AST and evaluates them to produce a curve $y(x)$.
   - **Goal:** Generate curves that fool the Student classifier into making incorrect operator predictions.

2. **The Student (CurveClassifierCNN):**
   - A 1D Convolutional Neural Network.
   - Ingests the 366-dim feature representation of the curve generated by the Teacher.
   - Predicts a multi-hot vector (9 classes) of which operators are present.
   - **Goal:** Accurately identify the operators used by the Teacher.

### 1.2 Teacher Rewards and Penalties

The Teacher receives a reward for each curve it generates based on:
$$ Reward = (Student\_BCE\_Loss) - (Complexity\_Penalty \times Length) $$

- **Student Loss:** Higher loss means the Teacher successfully tricked the Student.
- **Complexity Penalty:** Prevents the Teacher from generating infinitely long, meaningless mathematical noise. This penalty is smoothed by a *curriculum multiplier* tied to the Student's moving average loss.
- **Entropy Bonus:** Added to the Teacher's PPO/REINFORCE loss to prevent mode collapse (e.g., repeatedly generating just `x`).
- **Baseline Normalization:** Rewards are normalized across the batch (zero-mean) to stabilize gradients.
- **Invalid Sequences:** If the Teacher generates an invalid AST, causes a `NaN` via vertical asymptotes/singularities, or produces amplitudes $> 1e20$, the sequence is rejected and receives a fixed negative reward penalty.

### 1.3 Operator Classification & Vocabulary Space

The Teacher generates formulas built from a continuous vocabulary map which is then mapped into 9 structural classifier classes for the Student to predict:

**Teacher Generative Vocabulary:**
- Binary operators: `+`, `-`, `*`, `/`, `pow`
- Unary operators: `sin`, `cos`, `exp`, `log`, `sqrt`, `abs`
- Variables: `x`
- Mathematical Constants: `pi`, `e`
- Numeric Constants: `1.0`, `-1.0`, `0.5`, `2.0`

These raw tokens map into the 9 target classes for the Student:
- `identity` (from `x`)
- `sin`
- `cos`
- `power` (from `pow`, `sqrt`)
- `exp`
- `log`
- `addition` (from `+`, `-`)
- `multiplication` (from `*`)
- `rational` (from `/`)

### 1.4 Feature Extraction with Mult-SNR Noise

When the Teacher successfully generates a curve, the pipeline:
1. Augments the curve using `apply_noise_augmentation` with a 'multi'-SNR profile to ensure the Student learns robust features.
2. Extracts the standard 366-dimensional features (shape, FFT, derivatives, stats, curvature).
   - *Note:* Curvature gradients are heavily clipped to prevent `numpy` float64 overflows on asymptotes.

### 1.5 Student Training (Replay Buffer)

To prevent catastrophic forgetting or overfitting to the Teacher's most recent batch:
- Valid `(features, labels)` pairs are stored in a moving **Replay Buffer** (max size 2048).
- The Student samples mini-batches from this buffer and updates its weights using `BCEWithLogitsLoss`.
- Gradient clipping (`max_norm=1.0`) is enforced to prevent `NaN` crashes from extreme synthetic data.

## 3) Inference Stage (Curve Classifier Integration)

Primary script: scripts/curve_classifier_integration.py

### 3.1 Model loading

1. Resolve device (auto/cpu/cuda with fallback warning).
2. Cache model by key "device:model_path".
3. Load .pt (PyTorch) or .pkl/.joblib (XGBoost).
4. Load associated metadata (thresholds, temperature, scaler).

### 3.2 Single-input prediction flow

Input:
- x shape (N,) or (N,1)
- y shape (N,)

Flow:
1. Extract 366-d features from y.
2. Apply saved scaler if present.
3. Run classifier:
   - PyTorch: logits -> sigmoid (with optional temperature)
   - XGBoost: per-class predict_proba
4. Apply thresholds (per-class if saved, otherwise global threshold).
5. Build operator probability dictionary.
6. Add derived compatibility outputs:
   - periodic = max(sin, cos)
   - exponential = max(exp, log)
   - polynomial = max(power, identity)

### 3.3 Multi-input prediction flow (n_vars > 1)

Input:
- x shape (N, n_vars)
- y shape (N,)

Flow:
1. Build ND interpolator for y(x):
   - try LinearNDInterpolator
   - fallback NearestNDInterpolator
2. For each variable v:
   - fix all other variables at median values
   - sweep variable v across its observed min/max range
   - generate 1D slice y_v
   - remove NaN points
   - extract features and predict class probabilities
3. Aggregate across variables by max probability per class.
4. Threshold and return final operator dictionary + derived keys.

## 4) Fast-Path Direct Regression & Beam Search Stage

Primary script: `scripts/classifier_fast_path.py`

### 4.1 Trigger
When the classifier outputs probabilities for a curve, we run `run_guided_evolution(x, y, model_path, ...)`:
1. It first attempts the `fast_path_regression` to find an immediate symbolic approximation using LASSO.
2. If the fast path fails to find a high-quality solution, it falls back to `beam_search_evolution`.

### 4.2 Fast-Path Regression Strategy
1. **Basis Construction**: Construct a large, dense matrix of base functions using predicted operators (e.g., polynomials, $\sin(\omega x)$, $e^x$, $\log(|x|)$).
2. **Exact Search**: Do a combinatorial combinatorial exact match over basis pairs/triples (bounded by `exact_match_max_basis` to prevent explosions).
3. **LASSO**: If no exact match, run L1-regularized coordinate descent for feature selection.
4. **OLS Refit**: Take the sparse LASSO terms and refit them with simple Least Squares via `np.linalg.lstsq`.

### 4.3 Beam Search Evolution Fallback
If direct linear algebra fails, we generate diverse starting points for structural evolution:
- Initializes 20 "beams" (15 synthetically handcrafted from classifier priors, 5 purely random).
- Runs 3 rounds of short, parallel C++ evolution sequences.
- Retains the top 20% most diverse and physically accurate graphs across rounds.
- Feeds these winners into the main C++ Evolution loop as elite seeds.

## 5) C++ Evolutionary Search Stage

Primary core: `glassbox/sr/cpp/evolution.h` and `glassbox/sr/cpp/eval.h`

### 5.1 The C++ Engine (`EvolutionEngine`)
The main symbolic regression workhorse is now entirely written in C++ for extreme performance. Evaluates expressions fundamentally as `IndividualGraph` Directed Acyclic Graphs (DAGs) rather than PyTorch modules.

### 5.2 Age-Fitness Pareto Optimization (AFPO) & Selection
Evolution maintains diversity and prevents premature convergence by using a 3-objective **NSGA-II** selection tournament:
1. Minimize `MSE`
2. Minimize `Complexity` (number of graph nodes)
3. Minimize `Age` (surviving generations). Young graphs are protected while their continuous parameters mature.

### 5.3 Macro-Mutations
In addition to standard point mutations, the engine applies structural "Macro-Mutations" (15% probability) to preserve winning building blocks:
- **Wrap Mutation** (50%): $f(x) \rightarrow \sin(f(x))$ or $\exp(f(x))$
- **Multiply Mutation** (30%): Combines sub-graphs $f(x) \times g(x)$
- **Nest Mutation** (20%): Submits one sub-graph into another $f(g(x))$

### 5.4 Continuous Optimization & Snapping
Instead of PyTorch Autograd, continuous parameter discovery is handled internally:
1. **Local SGD:** Finite-difference Adam (`refine_inner_params`) tweaks frequencies, phases, and decay rates.
2. **Ridge Regression:** Evaluates linear combinations of graph terminal nodes to instantly solve output weights.
3. **In-Loop Snapping:** `cleanup_graph` actively snaps these floating-point parameters to exact integers, fractions, or constants ($\pi, e$) during the run if it degrades MSE by less than 1.01x.

## 6) End-to-End Answer Path (Concise Step List)

Step 1: The TeacherRNN generates a batch of synthetic formulas via autoregressive token sampling (`sin`, `x`, `pow`, etc.).

Step 2: Formulas are parsed into ASTs and evaluated on sampled $x$ domains to produce curves. Invalid/NaN curves receive a negative reward penalty.

Step 3: Curves are augmented with multi-SNR noise, and 366 features (shape, FFT, derivatives, stats, curvature) are extracted.

Step 4: The Teacher receives a reward based on how badly the StudentCNN misclassified the operators, minus a complexity penalty.

Step 5: The StudentCNN samples from a rolling Replay Buffer of Teacher-generated curves to learn robust 9-class operator detection.

Step 6: For a new test curve, extract features (or multi-input slices), run the `CurveClassifierCNN`, and produce operator probability thresholds.

Step 7: `classifier_fast_path.py` attempts a direct linear algebra solution via LASSO + OLS. If an exact match is found, immediately return.

Step 8: If Fast-Path fails to find the exact algebraic form, initialize the `beam_search_evolution` to discover diverse algebraic root topologies based on classifier hints.

Step 9: Pass the Beam Search survivors into the highly parallel C++ `EvolutionEngine`. NSGA-II applies Macro-Mutations and AFPO over thousands of generations to snap onto the final equation.

Step 10: Return discovered formula and fitness metrics (MSE/correlation), which is the final answer for symbolic regression.
